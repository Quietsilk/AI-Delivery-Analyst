## Контекст

После v1–v4 сервер остался монолитным: каждый UI-запрос полностью пересчитывал метрики, история хранилась только в localStorage браузера (теряется при очистке), не было фонового синка. Этот ТЗ превращает проект в production-ready инструмент с персистентным хранилищем и read-only UI.

---

## Ключевые архитектурные решения

### Инвариант 1 — UI read-only
Браузер **никогда** не инициирует расчёт метрик. Он только читает сохранённые снапшоты. Метрики считаются один раз при ingestion и сохраняются.

### Инвариант 2 — Иммутабельные снапшоты
Каждый запуск ingestion создаёт новую строку в SQLite. Никаких UPDATE/DELETE.

### Инвариант 3 — Throughput = дельта снапшотов
`throughput` = количество задач, resolved с момента предыдущего снапшота. Вычисляется в `ingestion.py`, не в `metrics.py`.

### Инвариант 4 — Period без пересчёта
`GET /history?period=30d` фильтрует строки SQLite по полю `timestamp`. Метрики не пересчитываются.

### Инвариант 5 — `calculate_metrics` без `period`
Функция `calculate_metrics(issues)` не принимает `cutoff`/`period` в сигнатуре.

# snapshots(id INTEGER PK, project_key TEXT, timestamp TEXT, metrics_json TEXT)

## Часть 2 — UI рефакторинг (STEP 7)

### Что удалить
- `CONFIG.telegramWebhookUrl`
- `runHistory` массив и `LS.KEYS.history`
- `LS.saveState()` — сохранение истории в localStorage
- `async function runReport()` — вызов legacy webhook
- `drawChart()` / `drawThroughputChart()` без аргументов

### Что добавить

**`refreshDashboard()`** — нажатие кнопки Refresh:
1. `GET /latest?project=KEY`
2. Если 404 → auto-trigger `POST /sync` + поллинг каждые 3s
3. Если 200 → `updateDashboard(snapshot.metrics)` + `updateDataAge(timestamp)` + `loadHistory()`

**`loadHistory()`** — загрузка истории для графиков:
1. `GET /history?project=KEY&period=X`
2. `drawChart(snapshots)` + `drawThroughputChart(snapshots)`

**`triggerSync()`** — кнопка "Sync" в сайдбаре:
1. `POST /sync { project, creds, jql }`
2. 202 → запускает поллинг → auto-refresh когда данные появятся

**`updateDataAge(timestamp)`** — показывает "Updated Xm ago" в статус-строке

**`drawChart(snapshots)` / `drawThroughputChart(snapshots)`** — принимают массив `{timestamp, metrics}` из API (не `runHistory`)

### Period picker
При смене периода вызывает только `loadHistory()` — KPI не меняются, только окно графиков.

### Auto-sync при 404
```
GET /latest → 404
    → POST /sync (тихо)
    → poll GET /latest каждые 3s (до 20 попыток = ~60s)
    → при 200 → refreshDashboard()
```

### LocalStorage после рефакторинга
`ada:runHistory` — **удалён**. История хранится в SQLite, читается через API.

## Acceptance Criteria

| # | Критерий |
|---|---------|
| 1 | `calculate_metrics(issues)` — нет параметра `period`/`cutoff` в сигнатуре |
| 2 | `POST /sync` возвращает `{ok, queued}` — без метрик в ответе |
| 3 | Снапшоты иммутабельны — в SQLite только INSERT, никогда UPDATE/DELETE |
| 4 | Throughput = кол-во resolved с `timestamp` предыдущего снапшота |
| 5 | `GET /history?period=30d` фильтрует по `timestamp` снапшота, не пересчитывает метрики |
| 6 | UI не содержит вызовов `calculate_metrics` — только HTTP-запросы к серверу |
| 7 | При 404 `/latest` UI автоматически запускает sync + поллит до появления данных |
| 8 | Кнопка "Sync" в сайдбаре → `POST /sync` → поллинг → auto-refresh |
| 9 | `drawChart` и `drawThroughputChart` принимают `snapshots` из API (не `runHistory`) |
| 10 | `ada:runHistory` отсутствует в localStorage — история только в SQLite |
| 11 | Фоновый планировщик запускается при наличии `PROJECTS` env |
| 12 | 108 тестов в 5 файлах, все проходят (`OK`) |
